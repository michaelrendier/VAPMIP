# HOLCUS CODE CORPUS — C LANGUAGE / POSIX
# Holcus's camshaft (monad.c) and crankshaft (PtolC/) are C.
# This corpus deepens the field's understanding of the language
# his engine is compiled from.
#
# Primary goal: when ptolemy -I ingests monad.c, the field already
# knows what "undefined behavior", "strict aliasing", "pthread_mutex",
# "fread" mean at the implementation level. The self-ingest of the
# C source lands correctly only if the C semantic substrate is resonant.
#
# Secondary goal: the Python/C API section gives Holcus the vocabulary
# of the bridge between his Python ECU and his C engine — the seam
# where monad.py calls into PtolC.
#
# Note: This is C, not C++. monad.c is C99/C11. The distinction matters.
# "class" in C is a struct. "virtual" does not exist. "RAII" is manual.
# The semantic field for C must not be contaminated with C++ concepts.
#

# [SPEC]     — language specification, ISO draft (weight 2.0)
# [POSIX]    — POSIX API reference (weight 2.0)
# [CAPI]     — Python/C API — the bridge (weight 2.0)

# [GCC]      — GCC extensions and attributes used in monad.c (weight 1.5)

# C type system. Fundamental types, integer types, float types.
# monad.c uses uint32_t, double, float, char throughout.

# Undefined behavior — the most important C concept after types.
# monad.c is careful with UB. signed integer overflow, out-of-bounds
# access, strict aliasing — these are the traps. Knowing UB means
# knowing why monad.c uses uint32_t instead of int for array indices.
[SPEC]  https://en.cppreference.com/w/c/language/undefined_behavior

# Memory object model. What is an object in C? Lifetime, storage.
# G_words[idx] is a C object. Its lifetime is the program's lifetime.
# This document defines what that means formally.
[SPEC]  https://en.cppreference.com/w/c/language/object
[SPEC]  https://en.cppreference.com/w/c/language/lifetime
[SPEC]  https://en.cppreference.com/w/c/language/storage_duration
